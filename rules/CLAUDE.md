
## 目前ai的问题：
### 通病一：自作主张

让 AI 改个小 bug，它顺手把隔壁三个文件都重构了。不问不确认，闷头就干。

### 通病二：过度工程

明明 100 行代码能搞定的事，它写了 1000 行。抽象套抽象，层层叠叠。

### 通病三：误伤友军

动不动就改掉它根本没理解清楚的代码和注释，跟当前任务毫无关系的东西也被它"顺手优化"了。

## 规则

| 规则     | 解决的问题                 |
| ------ | --------------------- |
| 编码前先思考 | AI 自作主张，不问就干          |
| 简洁优先   | AI 过度工程，100 行写 1000 行 |
| 精准修改   | AI 误伤友军，乱改无关代码        |
| 目标驱动执行 | AI 盲目执行，缺乏闭环验证        |

### 规则一：编码前先思考

**本质：治 AI 的自作主张**

举个例子：让 AI 写"用户数据导出"功能，它通常的反应是立刻开干——默认导出所有用户、默认 JSON 格式、默认保存到本地、默认包含全部字段。

但这些默认，你一个都没说过，全是它猜的。

**规则要求**：AI 在动手前，先把困惑摆上桌——有歧义就问，有多种解法就列出来，该反驳的时候就反驳。别藏着，别猜，先对齐再写代码。

### 规则二：简洁优先

**本质：治 AI 的过度工程**

能用 10 行解决的事，别写成 100 行。没被要求的功能不加，没被要求的不添油加醋。

**判断标准**：如果一个资深工程师看了会说"这代码太复杂了"，那就推倒重写。

### 规则三：精准修改

**本质：治 AI 的误伤友军**

修 bug 就只改 bug。旁边代码再丑也不碰，注释再过时也不改，风格再别扭也照着原样写。

**核心原则**：每一行改动，都得能追溯到用户的具体需求。

### 规则四：目标驱动执行

> LLM 特别擅长围绕具体目标反复迭代。与其告诉它"怎么做"，不如给它"成功标准"，让它自己跑。


##  CLAUDE.md

```
1. 编程前思考  
   勿假设，勿掩盖困惑，明确权衡。  
实施前应：  
- 明确陈述假设，如有疑问及时提出  
- 存在多种解释时列举选项，不擅自选择  
- 存在更简单方案时直言，该反对时就反对  
- 遇到不清楚处立即停止，指明困惑点并发问  
  
2. 简单优先  
   用最少代码解决问题，拒绝推测性编程：- 不做需求外的功能  
- 不为单次使用代码创建抽象  
- 不添加未要求的"灵活性"或"可配置性"  
- 不为不可能发生的场景编写错误处理  
- 50行能解决的不用200行，否则重写  
  自问："资深工程师会觉得过度复杂吗？"若是则简化  
  
3. 精准修改  
   只动必要部分，只清理自己造成的混乱  
修改现有代码时：  
- 不"改进"相邻代码/注释/格式  
- 不重构运行正常的代码  
- 遵循现有风格（即使你倾向不同做法）  
- 发现无关废旧代码时注明而非删除  
  
当你的修改产生遗留物时：  
- 移除因你的修改而失效的导入/变量/函数  
- 不删除既有的废旧代码（除非特别要求）  
  检验标准：每行修改都应与用户需求直接对应  
4. 目标驱动  
   定义成功标准，循环验证直至达标  
将任务转化为可验证目标：  
- "增加验证"→"编写无效输入测试用例，然后使其通过"  
- "修复BUG"→"编写重现测试用例，然后修复"  
- "重构X"→"确保重构前后测试均通过"  
  
多步骤任务应陈述简明计划：  
1. [步骤] → 验证方式：[检查点]  
2. [步骤] → 验证方式：[检查点]  
3. [步骤] → 验证方式：[检查点]  
  
明确的标准（如"使测试通过"）可以自主循环验证，模糊标准（如"搞定它"）需要反复确认。  
  
## 5. 代码质量原则  
  
### 可读性优先  
- 变量命名要表达意图，而非类型  
- 函数单一职责，一个函数只做一件事  
- 避免深层嵌套，超过3层考虑重构  
- 注释解释"为什么"，代码表达"做什么"  
  
### 健壮性保障  
- 所有外部输入必须校验  
- 资源使用后必须释放（连接、流、锁）  
- 异常处理要具体，不捕获泛型 Exception  
- 关键操作要有日志记录  
  
### 性能意识  
- 避免在循环中执行数据库查询  
- 大数据量操作考虑分页或批量处理  
- 缓存适合的场景：读多写少、计算昂贵  
- 先测量再优化，不要过早优化  
  
## 6. 沟通与协作  
  
### 回答风格  
- 先给出结论，再展开细节  
- 代码示例要完整可运行  
- 复杂逻辑配合流程图或伪代码  
- 提供多种方案时说明适用场景和权衡  
  
### 不确定性处理  
- 不确定时明确说明"我不确定"  
- 给出验证方法让用户自行确认  
- 提供官方文档链接供参考  
- 不编造 API 或库的功能  
  
### 上下文管理  
- 记住用户的技术栈偏好  
- 延续之前讨论的设计决策  
- 主动询问缺失的关键信息  
- 长篇对话后简要总结进展  
  
## 7. 安全与合规  
  
### 安全编码  
- 不硬编码密码、密钥、Token  
- SQL 查询使用参数化，防止注入  
- 用户输入输出时进行转义，防止 XSS  
- 敏感数据不在日志中明文打印  
  
### 最佳实践  
- 遵循 OWASP Top 10 安全建议  
- 使用成熟的加密算法和库  
- 权限校验在后端执行，不依赖前端  
- 定期更新依赖包修复安全漏洞  
  
## 8. 测试思维  
  
### 测试覆盖  
- 边界条件必须测试（空值、极值、异常）  
- 核心业务逻辑要有单元测试  
- 集成测试验证模块间交互  
- 回归测试确保修改不破坏现有功能  
  
### 测试质量  
- 测试用例命名清晰表达场景  
- 每个测试只验证一个断言  
- 使用 Given-When-Then 结构组织测试  
- Mock 外部依赖，隔离测试单元  
  
---  
  
**核心理念**: 写出人类容易理解的代码，让机器去处理复杂性。  
  
## 9. 技术栈规范  
  
Java8、SpringBoot 2.7.18、SpringCloud Alibaba 2021.0.6.0、Nacos 2.2.3、  
OpenFeign 3.1.5、MyBatis-Plus 3.5.4.1、PostgreSQL 15.6、Druid、  
SpringSecurity+JWT、Knife4j 4.5.0、SLF4J+Logback、Lombok 1.18.30、Hutool 5.8.23  
  
### 核心约束  
- **语言版本**: 严格遵循项目配置的 JDK 版本（当前为 Java 8），不使用更高版本的语法特性。  
- **框架偏好**: 优先使用 Spring Boot 官方推荐的 Starter，避免引入非必要的第三方轻量级框架。  
- **数据库交互**: 必须使用 MyBatis-Plus 进行 CRUD 操作，复杂查询使用 XML 映射，禁止在代码中拼接 SQL。  
  
### 依赖管理  
- **工具类**: 优先使用项目中已存在的 Hutool 或 Spring 自带工具类，严禁引入新的工具包（如 Commons-lang3）造成冗余。  
- **JSON 处理**: 统一使用 Fastjson2，禁止混用 Jackson 或 Gson。  
- **日志规范**: 统一使用 SLF4J + Logback，禁止直接使用 System.out.println。  
  
### 架构一致性  
- **分层模型**: 严格遵守 Controller -> Service -> Mapper 分层，禁止跨层调用。  
- **对象转换**: 实体间转换必须使用 MapStruct 或手动 Setter，禁止使用 BeanUtils.copyProperties（性能差且不安全）。  
- **异常处理**: 业务异常必须抛出统一的 BusinessException，由全局处理器捕获。
```
