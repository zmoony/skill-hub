# 业务流程编排器

本技能用于指导业务侧代码的完整开发流程，确保从需求到代码生成的每一步都经过确认和验证。通过标准化的6步流程，保证项目质量和客户满意度。

## 何时使用

- 用户需要创建新的业务模块
- 用户希望按照标准流程进行开发
- 用户需要从原型到代码的完整指导
- 用户要求每步都经过确认后再继续

## 流程概览

```
┌─────────────────────────────────────────────────────────┐
│                   业务流程编排                           │
├─────────────────────────────────────────────────────────┤
│  第1步：填写原型地址                                     │
│  第2步：填写需求PRD文件                                  │
│  第3步：业务接口文档地址                                 │
│  第4步：初始化脚本脚本                                   │
│  第5步：开始编码代码（使用java-business-module-gen）     │
│  第6步：启动项目，保证项目正常运行                       │
└─────────────────────────────────────────────────────────┘
```

## 详细流程

### 第1步：填写原型地址

**目标**：收集产品原型文档地址，了解界面设计和交互流程

**操作**：
1. 询问用户提供原型文档地址（如：Axure、墨刀、Figma等链接）
2. 验证链接是否可访问
3. 记录原型文档地址

**检查项**：
- [ ] 原型地址是否有效
- [ ] 原型是否包含完整的页面设计
- [ ] 原型是否包含交互说明

**输出**：
```
原型地址：[用户提供的链接]
状态：已确认
```

**客户确认**：
```
请确认原型地址是否正确：[链接]
确认后回复"确认"继续下一步，或提供新的地址。
```

---

### 第2步：填写需求PRD文件

**目标**：收集产品需求文档（PRD），了解业务需求和功能规格

**操作**：
1. 询问用户提供PRD文件路径或链接
2. 检查文件是否存在且可访问
3. 解析PRD内容，提取关键需求点

**检查项**：
- [ ] PRD文件是否存在
- [ ] PRD是否包含完整的功能描述
- [ ] PRD是否包含业务规则
- [ ] PRD是否包含数据字典

**如果文件不存在**：
```
⚠️ 提示：未找到PRD文件
请先完成以下操作：
1. 创建PRD文档（可参考项目模板）
2. 将文件放置在：wva-cloud/docs/prd/ 目录
3. 或提供在线文档链接

PRD模板参考：wva-cloud/agents/产品Agent.md
```

**输出**：
```
PRD文件：[文件路径或链接]
状态：已确认
关键需求：[提取的需求列表]
```

**客户确认**：
```
请确认PRD内容是否完整：
- 功能描述：[列出]
- 业务规则：[列出]
- 数据要求：[列出]

确认后回复"确认"继续下一步，或补充缺失内容。
```

### 第3步：业务接口文档地址

**目标**：收集或生成接口文档，使用 planning-with-files 进行任务细分，持久化任务状态支持跨会话恢复

**操作**：
1. 询问用户是否已有接口文档
2. 如果已有，读取文档地址
3. **核心步骤**：使用 planning-with-files 解析接口文档，生成细分任务清单
4. 将任务清单持久化到项目文件中

**任务文件位置**：
```
wva-cloud/plans/{module-name}/interface-tasks.md
```

**任务文件格式**：
```markdown
---
module: {模块名称}
created: {创建时间}
status: in_progress
---

# 接口实现任务清单

## 任务概览
- 总任务数：{N}
- 已完成：0
- 进行中：0
- 待处理：{N}

## 任务列表

### TASK-001: {接口名称}
- **状态**：pending
- **接口路径**：{API路径}
- **请求方法**：{GET/POST/PUT/DELETE}
- **实现内容**：
  - [ ] Entity 字段定义
  - [ ] DTO 请求/响应对象
  - [ ] Mapper 数据访问
  - [ ] Service 业务逻辑
  - [ ] Controller 接口实现
- **完成时间**：-
- **备注**：-

### TASK-002: {接口名称}
...
```

**planning-with-files 使用流程**：
```
1. 读取接口文档（Markdown格式）
2. 解析每个接口定义：
   - 接口路径
   - 请求方法
   - 请求参数
   - 响应格式
   - 权限要求
3. 为每个接口生成实现任务
4. 写入任务文件到 plans/{module-name}/interface-tasks.md
5. 展示任务清单给用户确认
```

**检查项**：
- [ ] 接口文档是否存在
- [ ] 接口文档是否包含完整的API定义
- [ ] 任务文件是否已生成
- [ ] 任务清单是否完整覆盖所有接口

**如果文档不存在**：
```
⚠️ 提示：未找到接口文档
将根据PRD自动生成接口文档草案，包含：
- REST API路径设计
- 请求参数定义
- 响应格式定义
- 权限控制要求

生成后请确认是否需要调整。
```

**输出**：
```
接口文档：[文档地址]
任务文件：wva-cloud/plans/{module-name}/interface-tasks.md
状态：任务已细分
API数量：[统计]
任务数量：[统计]
```

**任务执行流程**：
```
每个任务按以下顺序执行：
1. 将任务状态更新为 in_progress
2. 实现具体代码（Entity → DTO → Mapper → Service → Controller）
3. 完成后将任务状态更新为 completed
4. 记录完成时间
```

**跨会话恢复机制**：
```
新会话启动时：
1. 检查 plans/{module-name}/interface-tasks.md 是否存在
2. 读取任务清单状态
3. 统计已完成/待处理任务
4. 继续执行第一个 pending 任务

恢复提示：
```
📋 任务恢复检测
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
检测到已有任务文件：interface-tasks.md
模块：{module-name}
进度：
- 已完成：{N} 个任务
- 待处理：{M} 个任务
- 当前任务：TASK-{X}

是否继续执行未完成的任务？
回复"继续"开始执行，或"重新开始"清空任务。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**客户确认**：
```
请确认接口文档和任务清单是否符合要求：
- API路径：[列出]
- 任务数量：[统计]
- 任务划分：[列出每个任务]

确认后回复"确认"继续下一步，或提出修改意见。
回复"开始执行"可直接进入任务执行阶段。
```

### 第4步：初始化脚本脚本

**目标**：准备数据库初始化脚本，包含表结构和初始数据

**操作**：
1. 根据PRD和接口文档，生成数据库DDL脚本
2. 检查脚本是否符合项目规范
3. 验证脚本的可执行性

**检查项**：
- [ ] 表名是否使用 wva_yangzhong_ 前缀
- [ ] 是否包含所有公共字段（id, create_time, update_time等）
- [ ] 是否包含索引和约束
- [ ] 是否包含初始数据（如字典数据）

**脚本规范**：
```sql
-- 表名规范
CREATE TABLE wva_yangzhong.table_name (
    id BIGINT PRIMARY KEY,
    -- 业务字段
    field_name VARCHAR(100),
    -- 公共字段
    create_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    update_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    create_by BIGINT,
    update_by BIGINT,
    is_deleted SMALLINT DEFAULT 0,
    remark VARCHAR(500)
);

-- 索引
CREATE INDEX idx_table_name_field ON wva_yangzhong.table_name(field_name);
```

**如果脚本不存在**：
```
⚠️ 提示：未找到初始化脚本
将根据需求自动生成数据库脚本，包含：
- 表结构定义（DDL）
- 索引和约束
- 初始数据（DML）

生成后请确认是否需要调整。
```

**输出**：
```
初始化脚本：[脚本路径]
状态：已确认
表数量：[统计]
```

**客户确认**：
```
请确认数据库脚本是否符合要求：
- 表结构：[列出]
- 字段定义：[列出]
- 索引设计：[列出]

确认后回复"确认"继续下一步，或提出修改意见。
```

### 第5步：开始编码代码

**目标**：按照任务清单逐个实现接口，完成后更新任务状态

**操作**：
1. 确认前4步都已完成并得到客户确认
2. **读取任务文件**：`plans/{module-name}/interface-tasks.md`
3. **按任务顺序执行**：从第一个 pending 任务开始
4. **逐任务实现**：调用 `java-business-module-gen` 技能
5. **更新任务状态**：完成后标记为 completed

**前置检查**：
```

- [ ] 任务文件存在：已检查 ✓

```

cd wva-business-service

## 流程控制规则

### 顺序执行

流程必须按照 1→2→3→4→5→6 的顺序执行，不可跳过。

### 每步确认

每一步完成后，必须等待客户确认"确认"后，才能进入下一步。

### 中断处理

如果客户在任何一步提出修改意见：
1. 记录修改意见
2. 完成修改
3. 重新确认
4. 客户确认后再继续

### 前置条件检查

在执行第5步（代码生成）前，必须检查：
```
if (步骤1未完成) {
    提示："请先完成第1步：填写原型地址"
    return;
}
if (步骤2未完成) {
    提示："请先完成第2步：填写需求PRD文件"
    return;
}
if (步骤3未完成) {
    提示："请先完成第3步：业务接口文档地址"
    return;
}
if (步骤4未完成) {
    提示："请先完成第4步：初始化脚本脚本"
    return;
}
```

### 文件不存在处理

如果任何步骤需要的文件不存在：
1. 明确提示用户文件缺失
2. 提供文件创建指导
3. 等待用户完成后再继续
4. 不可自动生成（除非用户授权）

## 注意事项

1. **严格顺序**：必须按照1→2→3→4→5→6的顺序执行
2. **客户确认**：每步必须等待客户确认后才能继续
3. **文件检查**：文件不存在时明确提示，不可自动生成
4. **错误处理**：任何步骤失败时，暂停流程并提示错误
5. **状态跟踪**：实时更新流程状态，保持透明度
6. **修改记录**：记录所有修改意见和处理结果
7. **完成标志**：所有步骤完成后，显示完成标志和访问地址
8. **任务持久化**：任务文件存储在 `plans/{module-name}/` 目录，支持跨会话恢复
9. **恢复检测**：新会话启动时自动检测进行中的流程，提供恢复选项
10. **任务粒度**：每个接口作为一个独立任务，可单独执行和跟踪进度
