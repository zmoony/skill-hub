# 增补需求统一记录器

这个技能用于在多次迭代中持续记录“增补需求”，避免信息散落到不同文件，保证后续可追踪、可回溯。

## 何时使用

- 用户说“记录一下这次增补需求”
- 用户说“把这个需求补充写入变更记录”
- 用户说“追加到 change log”
- 用户提出“前端增补建议也要落文档”
- 用户要求把涉及改动模块、文件统一归档

## 目标输出

- 在当前工程中定位或创建统一记录文件：
  - `doc/change-log/change-log.md` 或 `docs/change-log/change-log.md`
- 追加一条结构化记录（追加到末尾，不改历史）
- 返回实际写入路径和本次记录摘要

## 执行步骤

按以下顺序执行，不要跳步：

1. 确认当前工程根目录（以当前工作目录项目根为准）
2. 选择记录根目录：
   - 若存在 `doc`，使用 `doc`
   - 否则若存在 `docs`，使用 `docs`
   - 若都不存在，创建 `docs`
3. 确保专有目录存在：`<root-doc-dir>/change-log`
4. 目标文件固定为：`<root-doc-dir>/change-log/change-log.md`
5. 若文件不存在，先初始化标题和说明
6. 根据模板组织本次增补条目
7. 将条目追加到文件末尾（正序时间线）
8. 回读文件末尾，确认写入成功
9. 返回写入结果（路径 + 时间 + 摘要 + 待补充项）

## 记录模板（严格使用）

```md
## [{{now:YYYY-MM-DD HH:mm}}] {{增补标题}}

### 1. 变更内容
- {{变更点1}}
- {{变更点2}}

### 2. 涉及的修改（模块）
- {{后端服务/前端页面/数据库/接口文档/...}}

### 3. 涉及的修改（文件）
- {{相对路径1}}
- {{相对路径2}}

### 4. 前端增补建议
- {{页面交互或字段展示建议}}

### 5. 备注（可选）
- {{无则写: 待补充}}

---
```

## 缺失信息策略

- 不阻塞写入。
- 用户未提供的字段统一写 `待补充`。
- 在回复中明确列出哪些字段待补充，便于下一轮补全。

## 写入约束

- 只能追加，不覆盖历史。
- 默认不去修改最后一条；仅当用户明确说“覆盖刚才那条”时，才允许改最后一条。
- 文件路径尽量使用相对当前工程根的路径，避免机器相关绝对路径污染记录。

## 初始化文件建议内容

当 `change-log.md` 不存在时，先写入：

```md
# Change Log - 增补需求统一记录

> 本文件用于沉淀开发过程中的增补需求，按时间正序追加，便于回溯。
> 记录项包含：时间、变更内容、涉及修改（模块+文件）、前端增补建议。

---
```

## 失败兜底

- 无权限写入：返回清晰报错并指出目标目录。
- 文件内容异常或编码异常：保留原文件，写入到 `change-log.recover.md` 并提示用户人工合并。

## 回复格式

每次写入后，回复应包含：

1. `记录成功` 或 `记录失败`
2. 实际文件路径
3. 本次记录时间
4. 本次增补标题
5. 待补充字段（如有）
